Guia de Comandos Git

Pré-requisitos e Configuração

git clone [URL]

Comando: git clone https://github.com/seu_id_de_usuario_github/fiplan

Resposta Esperada: Cria uma cópia local do repositório remoto na sua máquina, incluindo todas as branches e histórico de commits.

git config --local user.name "Seu nome"

Comando: git config --local user.name "Seu nome para exibição no commit"

Resposta Esperada: Define o nome do autor que será utilizado nos commits deste repositório.

git config --local user.email "Seu email"

Comando: git config --local user.email "seu_email_cadastrado_no_github"

Resposta Esperada: Define o email do autor que será associado aos commits neste repositório.

git config --global alias.lol

Comando: git config --global alias.lol "log --oneline --graph"

Resposta Esperada: Cria um atalho global (lol) para o comando git log --oneline --graph, facilitando a visualização do histórico de commits de forma concisa.

git config --list --show-origin

Comando: git config --list --show-origin

Resposta Esperada: Exibe todas as configurações Git aplicadas, mostrando de qual arquivo (global, local, etc.) cada configuração foi lida.

Reset

Entenda o ciclo de vida dos arquivos no Git: WORKING DIRECTORY (diretório de trabalho) → STAGE (área de preparação) → REPO (repositório).

O comando git reset varia seu comportamento dependendo do modo utilizado (soft, mixed, hard).

git reset --soft

Comando: git reset --soft HEAD~1

Descrição: Altera a branch para o commit especificado (neste caso, o commit anterior a HEAD), mas mantém as alterações no STAGE e no WORKING DIRECTORY.

git reset --mixed (padrão)

Comando: git reset --mixed HEAD~1

Descrição: Altera a branch para o commit especificado, move as alterações para o WORKING DIRECTORY e limpa a STAGE. É o comportamento padrão do git reset.

git reset --hard

Comando: git reset --hard HEAD~1

Descrição: Altera a branch para o commit especificado e descarta todas as alterações no WORKING DIRECTORY e na STAGE. Esta é uma ação destrutiva!

Outros Comandos Úteis

Revert

O git revert é usado para desfazer um commit específico, criando um novo commit que reverte as alterações do commit original. É uma ação não destrutiva e segura para branches compartilhadas.

git revert [commit]

Comando: git revert commit_que_esta_errado

Resposta Esperada: Cria um novo commit que desfaz as alterações do commit problemático. Você será solicitado a editar a mensagem do novo commit.

Cherry Pick

O git cherry-pick copia um ou mais commits de uma branch para outra, criando novos commits com as mesmas alterações.

git cherry-pick [commit]

Comando: git cherry-pick commit_hash

Resposta Esperada: Cria um novo commit na branch atual, aplicando as mudanças do commit_hash.

Stash

O git stash é um recurso para salvar temporariamente as alterações não "commitadas" (tanto no Working Directory quanto no Stage), permitindo que você mude de branch ou realize outras tarefas sem perder o seu progresso.

git stash save "descrição"

Comando: git stash save "descrição"

Resposta Esperada: "Saved working directory and index state...". Suas alterações são salvas em um "stash" e o seu diretório de trabalho fica limpo.

git stash pop

Comando: git stash pop

Resposta Esperada: "On branch...". As alterações são aplicadas de volta no seu diretório de trabalho e o "stash" é removido.

Log e Log Formatação

O git log permite formatar e filtrar o histórico de commits do seu repositório.

git log --oneline --stat

Comando: git log --oneline --stat

Resposta Esperada: Exibe o histórico de commits de forma concisa, mostrando o hash, a mensagem e um resumo das alterações (quais arquivos foram modificados).

Bisect

O git bisect é uma ferramenta poderosa para encontrar qual commit introduziu um bug, navegando de forma automática entre commits "bons" e "ruins".

git bisect

Comandos:

  • git bisect start: Inicia a busca.
  • git bisect bad: Marca o commit atual como "ruim" (com o bug).
  • git bisect good [commit_hash]: Marca o commit atual (ou um específico) como "bom" (sem o bug).
  • git bisect reset: Finaliza a busca.

Resposta Esperada: O Git irá navegar para um commit intermediário. Você deverá testar o código e usar git bisect good ou git bisect bad até que o Git reporte qual commit introduziu o bug.
